# Архитектура специализированных субагентов с изолированным контекстом

## Концепция

### Проблема
При глобальной установке скиллов (навыков) в Claude Code они загружаются в контекст всех агентов, включая координатора. Это расходует контекстное окно и создает шум — координатор видит знания, которые ему не нужны.

### Решение
Кастомный Claude Code плагин с архитектурой "координатор + специалисты":
- **Координатор** — легкий slash command (`/dev`), знает только имена агентов и workflow
- **Специалисты** — субагенты с ограниченными инструментами и изолированным контекстом
- **Скиллы** — глубокие знания, которые инжектируются **только** в агентов, работающих с релевантными файлами

### Принцип изоляции (почему это работает)
1. Каждый субагент получает **чистый контекст** при создании — не наследует сессию координатора
2. Скиллы инжектируются через хуки PreToolUse/UserPromptSubmit **по паттернам файлов**
3. Каждый субагент получает свежий dedup-state — скиллы инжектируются независимо
4. Координатор не читает файлы проекта напрямую → скиллы специалистов в него не попадают

## Структура плагина

```
dev-team/
├── .claude-plugin/
│   └── plugin.json                    # Манифест плагина
├── commands/
│   └── dev.md                         # Координатор (slash command /dev)
├── agents/
│   ├── _template.md                   # Шаблон для создания новых агентов
│   └── code-reviewer.md              # Пример: ревьюер (read-only)
├── skills/
│   └── _template/
│       ├── SKILL.md                   # Шаблон скилла с примером metadata
│       └── references/
│           └── _template.md           # Шаблон reference-файла
└── CLAUDE.md                          # Инструкции плагина
```

## Компоненты плагина

### 1. plugin.json — манифест

```json
{
  "name": "dev-team",
  "description": "Framework for specialized development agents with isolated contexts",
  "agents": ["./agents"],
  "commands": ["./commands"],
  "skills": "./skills"
}
```

- `agents` — директория с .md файлами агентов (автообнаружение)
- `commands` — директория со slash commands (автообнаружение)
- `skills` — директория со скиллами (инжектируются по паттернам, НЕ все сразу)

### 2. Координатор (`/dev`) — slash command

Координатор — легкая команда, которая управляет workflow:

```yaml
---
description: Координирует работу специализированных агентов для разработки
argument-hint: Описание задачи или фичи
allowed-tools: Bash(git status), Bash(git diff:*), Bash(git log:*), Read, Glob, Grep
---
```

**5-фазный workflow координатора:**

| Фаза | Действие | Детали |
|------|----------|--------|
| 1. Анализ | Понять задачу | Определить нужных специалистов, декомпозировать |
| 2. Dispatch | Запустить агентов | Параллельно если независимы, последовательно если зависимы |
| 3. Сбор | Обработать результаты | DONE/BLOCKED/NEEDS_CONTEXT → решить что дальше |
| 4. Ревью | Code review | Отправить на code-reviewer (опционально) |
| 5. Отчёт | Сводка для user | Что сделано, файлы, тесты |

**Принципы dispatch (формирование prompt для субагента):**
- Вставлять **полный текст задачи** (не ссылки на файлы — агент не видит контекст координатора)
- Указывать **границы ответственности** (scope: какие файлы/директории можно менять)
- Включать **контекст** (что уже сделано другими агентами, от чего зависит)
- Требовать **структурированный отчёт** по протоколу (см. ниже)
- Независимые задачи → несколько Agent tool calls **в одном сообщении** (параллельный dispatch)

### 3. Агенты-специалисты

Каждый агент — `.md` файл в `agents/` со следующей структурой:

```yaml
---
name: agent-name                    # kebab-case, 3-50 символов (обязательно)
description: >                      # Когда использовать (обязательно)
  Use this agent when [условия].
  
  <example>
  Context: [Сценарий]
  user: "[Запрос]"
  assistant: "[Как Claude использует агента]"
  <commentary>[Почему подходит]</commentary>
  </example>

model: sonnet                       # sonnet|opus|haiku|inherit (обязательно)
color: blue                         # blue|cyan|green|yellow|magenta|red (обязательно)
tools: ["Read", "Write", "Edit"]    # Ограничение инструментов (опционально, без = все)
---

[System prompt — роль, экспертиза, правила, протокол отчёта]
```

**Ограничение `tools:` по ролям:**

| Роль | tools | Зачем |
|------|-------|-------|
| Имплементатор | `Read, Write, Edit, Grep, Glob, Bash` | Полный доступ к файлам и терминалу |
| Ревьюер | `Read, Grep, Glob` | Только чтение — не может менять код |
| Планировщик | `Read, Grep, Glob` | Только анализ — планирует, не реализует |
| Тестировщик | `Read, Write, Edit, Grep, Glob, Bash` | Запуск тестов + написание тестов |
| Исследователь | `Read, Grep, Glob, WebSearch, WebFetch` | Чтение + веб-поиск |

**Выбор модели:**
- `haiku` — простые задачи (форматирование, линтинг, переименование)
- `sonnet` — стандартная разработка (имплементация, тесты, ревью)
- `opus` — архитектурные решения, сложный рефакторинг

### 4. Скиллы (динамические знания)

Скиллы — глубокие знания, которые **НЕ загружаются все сразу**. Они инжектируются по паттернам:

```yaml
---
name: skill-name
description: "Когда использовать"
metadata:
  priority: 6                         # 4-8 (выше = инжектируется раньше)
  pathPatterns:                        # При работе с этими файлами (PreToolUse)
    - "**/*.tsx"
    - "**/components/**"
  bashPatterns:                        # При запуске этих команд
    - "npm run dev"
  importPatterns:                      # При наличии этих импортов в коде
    - "react"
    - "next"
  promptSignals:                       # При наличии этих фраз в prompt (UserPromptSubmit)
    phrases: ["react component"]       # +6 баллов за совпадение
    allOf: [["component", "style"]]    # +4 балла за группу
    noneOf: ["backend only"]           # Жёсткое подавление
    minScore: 6                        # Порог для инъекции
---

[Содержание скилла — best practices, паттерны, правила]
```

**Механизм инъекции:**

| Триггер | Как работает | Бюджет |
|---------|-------------|--------|
| PreToolUse (Read/Write/Edit/Bash) | pathPatterns / bashPatterns совпадают с файлом/командой | ≤3 скилла (~18KB) |
| UserPromptSubmit | promptSignals.phrases совпадают с текстом prompt | ≤2 скилла (~8KB) |
| importPatterns | Паттерн найден в импортах читаемого файла | Входит в бюджет PreToolUse |

**Почему координатор не получает скиллы:**
- Координатор НЕ читает `.tsx` файлы → pathPatterns не срабатывают
- Координатор НЕ запускает `npm run dev` → bashPatterns не срабатывают
- Prompt координатора не содержит "react component" → promptSignals не срабатывают

**Progressive disclosure (для больших скиллов):**
```
skills/my-knowledge/
├── SKILL.md              # Краткая сводка (инжектируется в контекст)
└── references/           # Глубокие знания (агент читает по необходимости)
    ├── patterns.md
    ├── anti-patterns.md
    └── examples.md
```

## Протокол отчёта субагентов

Каждый агент завершает работу структурированным отчётом:

```
Status: DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT

Files changed: [список файлов]
Summary: [что было сделано]
Tests: [какие тесты написаны/запущены и результат]
Concerns: [если DONE_WITH_CONCERNS — что беспокоит]
Blocked on: [если BLOCKED — что мешает продолжить]
Questions: [если NEEDS_CONTEXT — какая информация нужна]
```

**Обработка координатором:**

| Статус | Действие координатора |
|--------|----------------------|
| DONE | Перейти к следующему этапу |
| DONE_WITH_CONCERNS | Прочитать concerns, решить нужно ли действовать |
| BLOCKED | Предоставить недостающий контекст, перезапустить агента |
| NEEDS_CONTEXT | Ответить на вопросы, перезапустить агента |

## Привязка скилл → агент

Привязка происходит **неявно** через паттерны, а не через прямой маппинг:

```
frontend-specialist работает с .tsx файлами
    → pathPattern "**/*.tsx" срабатывает
    → frontend-knowledge SKILL.md инжектируется

backend-specialist работает с .py файлами  
    → pathPattern "**/*.py" срабатывает
    → backend-knowledge SKILL.md инжектируется

code-reviewer читает любые файлы
    → соответствующие скиллы инжектируются в зависимости от типа файла
```

**Дополнительный контроль:** координатор может включать ключевые фразы из `promptSignals` в prompt агента, чтобы гарантировать инъекцию нужного скилла:
```
"Ты frontend specialist. Работай с react components и CSS стилями."
→ phrases ["react component"] → frontend-knowledge инжектируется
```

## Три уровня сложности (эволюция)

### Уровень 1: Agent-Only (простейший)
- Только markdown-файлы, без скриптов
- Знания вшиты прямо в тело каждого агента
- Нет директории skills/
- Подходит для старта с 2-3 агентами

### Уровень 2: Plugin + Skills (текущий план)
- Знания вынесены в skills/ с динамической инъекцией
- Агенты легкие, скиллы инжектируются по паттернам
- Координатор полностью изолирован от знаний специалистов

### Уровень 3: Полный с хуками (продвинутый)
- TypeScript хуки на SubagentStart для детерминированного маппинга агент→скиллы
- Хук SubagentStop для автоматического сбора отчётов
- Полный контроль над инъекцией через agent_id
- Требует компиляции TypeScript

## Установка и использование

### Установка

```bash
# Из локальной директории:
claude plugins add /path/to/dev-team

# Из git-репозитория:
claude plugins add https://github.com/user/dev-team
```

### Использование

```bash
# Запуск координатора с задачей:
/dev Реализовать систему аутентификации с JWT и OAuth2

# Координатор автоматически:
# 1. Анализирует задачу
# 2. Определяет нужных специалистов  
# 3. Dispatch'ит агентов (параллельно если можно)
# 4. Собирает результаты
# 5. Отправляет на ревью
# 6. Отчитывается
```

### Добавление нового агента

1. Скопировать `agents/_template.md` → `agents/my-specialist.md`
2. Заполнить frontmatter: name, description (с examples), model, color, tools
3. Написать system prompt с экспертизой
4. Добавить протокол отчёта в конец
5. Перезапустить Claude Code — агент обнаружится автоматически

### Добавление нового скилла

1. Создать директорию `skills/my-knowledge/`
2. Создать `skills/my-knowledge/SKILL.md` с YAML frontmatter:
   - `pathPatterns` — для каких файлов инжектировать
   - `promptSignals` — для каких фраз в prompt инжектировать
3. Опционально: создать `references/` для глубоких знаний
4. Перезапустить Claude Code — скилл обнаружится автоматически

## Верификация после установки

| Проверка | Как | Ожидаемый результат |
|----------|-----|---------------------|
| Плагин установлен | Ввести `/dev` | Команда доступна |
| Агенты доступны | Claude предлагает агентов | code-reviewer и другие в списке |
| Изоляция tools | Вызвать code-reviewer | Write/Edit недоступны |
| Инъекция скиллов | Агент читает .tsx → скилл есть | Скилл инжектирован |
| Чистота координатора | /dev не видит скиллы | Контекст координатора чистый |
| Параллельный dispatch | Дать 2+ независимые задачи | Агенты запущены параллельно |

Для отладки: `claude --debug` покажет инъекцию скиллов и работу хуков.
